# Migration guide

Atlas is adopting Mintlify as the long-term documentation surface for the project.

The API docs are the first slice because the repo already has a strong contract source of truth in `openapi/atlas.openapi.json`, and Mintlify can generate endpoint reference pages from a synced local copy of that file inside `mintlify/openapi/`.

## What Mintlify owns now

- The docs shell defined in `docs.json`
- The introductory Atlas pages in this root-level MDX structure
- The API overview pages under `api-reference/`
- The generated API reference driven from `openapi/atlas.openapi.json`

## What stays in `docs/` for now

The existing `docs/` tree still holds architecture, development, standards, deployment, and design material that has not been migrated yet.

Until those sections move, treat the current state like this:

1. Mintlify is the forward direction.
2. `docs/` remains valid background and contributor documentation.
3. Avoid duplicating stable API endpoint details in both places when the generated OpenAPI reference already covers them.

## Contributor workflow

Use these repo-level commands while working on the new docs surface:

```bash
pnpm docs:dev
pnpm docs:validate
pnpm docs:openapi:check
```

If the API contract changes, regenerate the spec before validating the docs:

```bash
pnpm openapi
pnpm docs:validate
```

`pnpm openapi` now refreshes both the root OpenAPI artifact and the Mintlify-local copy that the docs site reads.

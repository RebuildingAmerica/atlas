Atlas plans to publish to the [MCP Registry](https://registry.modelcontextprotocol.io) so registry-aware clients can find and connect to the server without operators wiring up the URL by hand.

<Note>
  Atlas is **not currently listed** in the MCP Registry. The earlier `1.0.0` listing was unpublished pending stabilization of the read/write tool surface; the next publish will start from `0.1.0`. The connection URL below remains live in the meantime.
</Note>

## Listing details (planned)

| Field | Value |
|-------|-------|
| Namespace | `org.rebuildingus.atlas/atlas` |
| Server URL | <McpServerUrl /> |
| Transport | `streamable-http` |
| Version | `0.1.0` |
| Authorization | OAuth 2.1 with PKCE, optional CIMD |

The namespace follows the [MCP Registry naming rules](https://modelcontextprotocol.io/registry/namespaces): reverse-DNS, scoped to a domain Atlas controls. Ownership of `rebuildingus.org` grants the entire `org.rebuildingus.*` prefix; the trailing `atlas` segment scopes this manifest within that prefix.

## How clients will discover Atlas

Once relisted, registry-aware clients can:

1. **Search by name or namespace.** A user types "Atlas" or `org.rebuildingus.atlas/atlas` into a registry-aware UI and the registry returns the server URL plus the OAuth discovery hints.
2. **Pre-flight the server.** Clients hit the listed URL, get back a `401` with a `WWW-Authenticate` header pointing at Atlas's [protected-resource metadata](https://atlas.rebuildingus.org/.well-known/oauth-protected-resource), and walk the OAuth 2.1 discovery chain from there.
3. **Onboard without a prior relationship.** New clients with no Atlas account use [Client ID Metadata Documents](/mcp/client-id-metadata) to register dynamically without an Atlas-side admin step.

The manual fallback works today regardless of registry state — point any MCP client directly at the URL:

```
claude mcp add atlas https://atlas.rebuildingus.org/mcp
```

For client-by-client setup walkthroughs, see the [MCP overview](/mcp/overview).

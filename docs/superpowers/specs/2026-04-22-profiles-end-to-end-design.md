# Profiles: End-to-End UI/UX Design

Date: 2026-04-22

## Context

Atlas profiles are the public face of the civic graph — the pages that answer "who is this person/organization and why should I trust what Atlas says about them?" The existing April 11 spec defined profile components (PersonProfile, OrgProfile) and they're implemented, but the current experience has critical gaps:

- Profiles are client-side rendered (`ssr: false`), invisible to search engines
- No Open Graph or structured data — shared links show nothing
- No canonical URLs — profiles live at `/entries/:id` (opaque UUIDs)
- No way to discover related actors from a profile — profiles are dead ends
- No network exploration surface

This spec defines the complete discovery-to-profile-to-exploration flow: how users find profiles, what they see, how they navigate between actors, and how profiles present to search engines and social platforms.

## Architecture: Profile + Explorer Split

The profiles feature spans two dedicated surfaces with distinct rendering strategies:

1. **Profile Page** — SSR, fast, SEO-optimized. A research document about a single actor.
2. **Network Explorer** — CSR, interactive. A graph visualization tool for exploring connections.

These are separate surfaces, not layered. The profile page links to the explorer; the explorer links back to profiles. Transitions between them are sequential — one surface exits, the next enters.

### Surface Map

## URL Structure

### Canonical Profile URLs

```
/profiles/people/:slug
/profiles/organizations/:slug
```

Slugs are autogenerated as `{name-slug}-{short-id}` (e.g., `jane-doe-a3f2`). The short ID is a 4-character lowercase alphanumeric hash derived from the entry UUID, providing collision safety.

Profile owners can claim their profile and set a vanity slug (e.g., `/profiles/people/janedoe`). When a vanity slug is set, the autogenerated slug redirects (301) to the vanity slug. Both resolve, but the vanity slug is canonical.

### Legacy Redirect

`/entries/:id` performs a server-side 301 redirect to the canonical profile URL. This preserves link equity from any existing links and teaches search engines the canonical path.

### Explorer URL

```
/explore?actor=:id
```

The explorer is not indexed (CSR-only, no SSR). The query parameter seeds the graph with the specified actor's neighborhood.

## Profile Page Design

### Rendering Strategy

Profile pages are server-side rendered. TanStack Start's SSR mode generates full HTML on the server, including meta tags and structured data.

**Cache strategy:** CDN-level caching with `stale-while-revalidate`. Core profile data (name, description, issues, org affiliation) is stable and can be cached for hours. Source counts and latest appearances are included in the SSR response but are acceptably stale — the pipeline doesn't update more than a few times per day.

### Layout

A centered max-width container (matching the existing `PageLayout` pattern) with two columns on desktop:

- **Main column (~60%):** Narrative content — About/Mission, Issue Focus/Footprint, Appearances & Mentions/Coverage
- **Sidebar (~40%):** Relational content — Organization (for people), People (for orgs), Reach/Presence, Connections, "Explore this network" CTA

On viewports below the breakpoint, the sidebar stacks below the main column.

The dark header spans the full width of the centered container. The two-column split begins in the cream body area below the header.

### Person Profile Sections

**Dark Header (full container width):**
- Type label ("Person"), verified badge, mention count
- Circular avatar (56px) with gradient initials fallback
- Bold name (22px/700, slight negative letter-spacing)
- Role subtitle (e.g., "Founder, Prairie Workers Cooperative")
- Location + geo-specificity pills

**Main Column:**
- **About** — 1-3 sentence description
- **Issue Focus** — Flat list of issue area pills (accent-colored)
- **Appearances & Mentions** — First appearance expanded with extraction context quote (left-bordered quote block). Remaining as compact list rows (source type badge + publication + date). Total count badge. "View all N appearances" link.

**Sidebar:**
- **Organization** — Linked card to affiliated org (square logo mark, name, location). Navigates to org profile on click.
- **Reach** — Stacked cards: email, website, phone (icon + label + value)
- **Connections** — Related actors as cards, grouped by relationship type with evidence snippets. See Connections Section below.
- **Explore this network** — CTA button linking to `/explore?actor=:id`

### Organization Profile Sections

**Dark Header (full container width):**
- Type label ("Organization"), verified badge, mention count
- Square logo mark (52px, rounded 12px corners) with gradient fallback
- Bold name (22px/700)
- Location + geo-specificity + active status pills

**Main Column:**
- **Mission** — Organization description
- **Issue Footprint** — Issue areas grouped by taxonomy domain. Domain name as muted label, issue area pills underneath. Domain-specific tint colors.
- **Appearances & Coverage** — Coverage breakdown bar (stacked horizontal bar, 6px height, color-coded by source type with legend). Then compact source list rows. Total count badge. "Browse all N appearances" link.

**Sidebar:**
- **People** — Avatar row (44px circular avatars with first-name labels). Click expands a detail panel: larger avatar, full name, role, mention count, bio snippet, issue tags, "View full profile" link.
- **Presence** — Prominent website card (32px dark icon container + domain + "Visit" pill). 2-column grid for email, phone, first-seen date.
- **Connections** — Related actors grouped by relationship type with evidence. See below.
- **Explore this network** — CTA button.

### Connections Section (Shared)

The connections section appears in the sidebar of both person and org profiles. It shows actors related to the current profile, grouped by relationship type:

**Relationship types:**
- **Same organization** — other people at the same org, or the org itself
- **Same issue area** — actors working on overlapping issue areas in the same geography
- **Same geography** — actors in the same city/region working on different issues
- **Co-mentioned** — actors mentioned in the same source

Each connection card shows:
- Actor avatar (circle for person, rounded square for org)
- Name and brief descriptor
- Relationship type label
- Evidence snippet (e.g., "Co-mentioned in: Kansas City Star, Mar 2026")

Cards link to the connected actor's profile page.

**Data source:** Connections are computed server-side from existing data:
- Same org: `affiliated_org_id` matching
- Same issue area: `entry_issue_areas` overlap + geographic proximity
- Same geography: location field matching
- Co-mentioned: shared entries in `entry_sources` junction table

## Network Explorer

### Purpose

A full-screen interactive graph canvas for exploring how actors connect. Entered from a profile's "Explore this network" CTA. Opens with the source actor as the center node and their 1-hop connections pre-loaded.

### Layout

Full-viewport canvas with a sidebar panel:

- **Canvas (full viewport):** Force-directed graph. Dark background (#0f0f0f). Nodes are actor avatars (circles for people, rounded squares for orgs). Edges connect related actors with color-coded relationship types. Zoom, pan, and drag supported.
- **Sidebar (right, ~320px):** Shows a mini-profile of the currently selected node — avatar, name, type, description snippet, issue tags, mention count, "Open full profile" link. When no node is selected, shows filter controls.

### Interactions

- **Click node:** Selects it, sidebar shows mini-profile
- **Double-click node:** Expands to load that node's connections (grows the graph)
- **Click edge:** Highlights relationship, sidebar shows evidence
- **Open full profile:** Navigates to the SSR profile page (explorer surface exits, profile enters)
- **Filter controls:** Filter visible nodes by issue area, geography, or entry type. Clustering by these dimensions.

### Rendering

Client-side only. No SSR, no indexing. The explorer fetches graph data from a dedicated API endpoint that returns nodes and edges for a given actor's neighborhood.

### Graph Library

Use a performant graph visualization library suited to the interaction requirements (force layout, zoom/pan, node expand). Candidates: D3-force with custom renderer, or a dedicated graph library. Exact choice deferred to implementation.

## Transitions

All page transitions use the View Transitions API via TanStack Router's `viewTransition` support. No CSS `transform: scale()` animations. No overlapping surfaces — transitions are sequential.

### Transition Vocabulary

Four motion primitives:

1. **Morph** — Shared elements (avatar, name) interpolate actual position and dimensions between views via `view-transition-name`. Size changes use real width/height, not scale transforms.
2. **Slide** — Profile-to-profile navigation: outgoing profile exits left, incoming profile enters from right. Lateral motion implies traversal through the graph.
3. **Fade** — Surface-level transitions (profile ↔ explorer): outgoing surface fades out, incoming fades in. Clean cut, no overlap.
4. **Stagger** — Graph nodes appear with 50ms stagger radiating from center. Profile sections appear with top-down stagger on initial load.

### Transition Map

| From | To | Motion |
|---|---|---|
| Browse → Profile | Morph (avatar + name from entry card to header) + Fade (browse surface out, profile in) |
| Profile → Profile | Slide (old left, new right) + Morph (avatar if same type) |
| Profile → Explorer | Fade (profile out, explorer in). Center node placed at avatar's last position. Graph nodes stagger in. |
| Explorer → Profile | Fade (explorer out, profile in). Avatar placed where the selected node was. |
| /entries/:id → Profile | Server-side 301 redirect (no client transition) |

### Fallback

Browsers without View Transitions API support get instant page swaps (no animation). No broken states — transitions are progressive enhancement.

## API Requirements

### Existing Endpoints (sufficient for profiles)

- `GET /api/entities/:id` — Full entry detail with sources (powers profile page)
- `GET /api/entities` — List/search with filters (powers browse)

### New Endpoints Needed

**Connections endpoint:**
```
GET /api/entities/:id/connections
```
Returns actors related to the given entry, grouped by relationship type, with evidence. Response shape:
```json
{
  "connections": [
    {
      "type": "same_organization",
      "actors": [
        {
          "id": "...",
          "name": "...",
          "type": "person",
          "slug": "...",
          "avatar_url": null,
          "description_snippet": "...",
          "evidence": "Affiliated via Prairie Workers Cooperative"
        }
      ]
    },
    {
      "type": "co_mentioned",
      "actors": [
        {
          "id": "...",
          "name": "...",
          "type": "person",
          "slug": "...",
          "evidence": "Both mentioned in: Kansas City Star, Mar 2026"
        }
      ]
    }
  ]
}
```

**Graph neighborhood endpoint:**
```
GET /api/entities/:id/graph?depth=1
```
Returns nodes and edges for the explorer. Response shape:
```json
{
  "nodes": [
    {
      "id": "...",
      "name": "...",
      "type": "person",
      "slug": "...",
      "issue_areas": ["housing", "labor"],
      "city": "Kansas City",
      "state": "MO"
    }
  ],
  "edges": [
    {
      "source": "...",
      "target": "...",
      "relationship": "same_organization",
      "evidence": "Both affiliated with Prairie Workers Cooperative"
    }
  ]
}
```

**Slug resolution endpoint:**
```
GET /api/entities/by-slug/:type/:slug
```
Resolves a type + slug pair to an entry. The `:type` parameter is `people` or `organizations` (matching the URL structure). Used by the SSR profile route to fetch data server-side. Returns the full entity response (same shape as `GET /api/entities/:id`). Returns 404 if no entry matches, or 301 with the canonical slug if the provided slug is an alias.

### Schema Changes

**New column on `entries` table:**
```sql
ALTER TABLE entries ADD COLUMN slug TEXT UNIQUE;
```

Slugs are generated on insert/update. Format: `{name-slug}-{short-id}` where short-id is the first 4 characters of a hash of the entry UUID. Vanity slugs (set by profile owners) override the autogenerated slug; the old slug is stored as an alias for redirect purposes.

**New table for slug aliases:**
```sql
CREATE TABLE slug_aliases (
  old_slug TEXT PRIMARY KEY,
  entry_id TEXT NOT NULL REFERENCES entries(id),
  created_at TEXT NOT NULL DEFAULT (datetime('now'))
);
```

When a vanity slug replaces an autogenerated slug, the old slug is inserted into `slug_aliases`. The route handler checks `slug_aliases` if the primary slug lookup fails, and issues a 301 redirect.
